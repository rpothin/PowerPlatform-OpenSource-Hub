## Remaining Priority Issues

### [P2] Empty state emoji has conflicting ARIA attributes
**What**: `<span role="img" aria-hidden>🔍</span>` — `role="img"` announces the element to screen readers as an image, but `aria-hidden="true"` tells screen readers to ignore it. These contradict each other.
**Why it matters**: Screen reader behavior is undefined for this combination — VoiceOver on Safari may announce it, NVDA on Windows may ignore it, neither is predictable.
**Fix**: Pick one. Since the emoji is decorative: remove `role="img"`, keep only `aria-hidden`. Done: `<span aria-hidden="true">🔍</span>`.
**→ `/impeccable audit`**

### [P2] No visibility feedback when filters are active
**What**: When a user has filters applied and sees "47 repositories found", there's no persistent visual summary of which filters are active in the gallery body area (above the results). The filter state lives in the left-pane accordion, which may be scrolled out of view.
**Why it matters**: Users who scroll down into results lose context of what's filtered. They can't tell at a glance why they're seeing fewer repos without scrolling back to the filter pane.
**Fix**: Add a compact active-filter tag row in the gallery header area (below the result count), showing each active filter as a dismissible pill. Each pill removes its filter on click.
**→ `/impeccable harden`**

### [P2] Search input has no clear affordance
**What**: The search bar on the gallery page has no visible clear button (×). To clear a search, the user must manually select all and delete.
**Why it matters**: This is a standard pattern for search inputs and its absence is a frustration point for all three personas. FluentUI's `Input` component supports a `contentAfter` prop for exactly this.
**Fix**: Add a dismiss icon button in `contentAfter` on the FluentUI `Input` when `filterState.searchText.length > 0`.
**→ `/impeccable harden`**

### [P2] Dead `.filterPaneAndGallery` CSS block in `index.module.css`
**What**: `index.module.css` still contains the `.filterPaneAndGallery`, `.heroBar`, `.buttons`, and `.searchBox` CSS classes (lines 249–289). These are no longer referenced by any component since `gallery.tsx` moved to `gallery.module.css`.
**Why it matters**: Dead CSS misleads future maintainers into thinking these classes are in use. Any editor without CSS module resolution will flag them as "used" because the names are vague.
**Fix**: Remove the dead block. It's safe — no component references these class names from `index.module.css` anymore.
**→ `/impeccable harden`**

### [P3] Filter accordions collapsed by default
**What**: All filter sections (Topics, Languages, Licenses, etc.) are collapsed by default. First-time users can't tell what's available without clicking every accordion.
**Why it matters**: Jordan (first-timer) lands on the gallery, sees a search bar and a "Filters" label, but no visible filter options. She has to actively discover that topics, languages, and licenses exist as facets.
**Fix**: Expand the top 3-4 most-used filter sections by default (Good First Issue, Topics, Languages — these are the highest-intent facets). Keep the rest collapsed.
**→ `/impeccable onboard`**

### [P3] No keyboard shortcut to focus search
**What**: No "/" or Cmd+K shortcut focuses the gallery search input.
**Why it matters**: Power users (Alex) reaching the gallery from a bookmark expect the "/" shortcut — it's a GitHub convention the audience knows cold.
**Fix**: Add a `useEffect` in `gallery.tsx` listening for `keydown` where `key === '/'` and `document.activeElement` is not already an input — then call `document.getElementById('filterBar')?.focus()`.
**→ `/impeccable harden`**

## Minor Observations

- The gallery hero subtitle (`rgba(255, 255, 255, 0.88)`) on the lighter gradient end (#8A60F9) is theoretically ~3.96:1 at the gradient's far edge. In practice, centered text sits over the gradient midpoint (#6936F7, ~5.85:1). Acceptable given reading area. No change required.
- `gallery.module.css` has a redundant dark-mode rule: `[data-theme="dark"] .galleryHero { background: linear-gradient(...) }` — identical to the light-mode rule (the hero gradient is the same in both modes; the --ds-hero-* tokens switch the underlying colors). Can reduce to just one rule. P3.
- The empty state `<Button appearance="primary">` uses FluentUI brand theming, so it will render in indigo-violet. Correct and on-brand.
- The `<span aria-hidden>🔍</span>` fix (P2 above) is a 5-second edit.
