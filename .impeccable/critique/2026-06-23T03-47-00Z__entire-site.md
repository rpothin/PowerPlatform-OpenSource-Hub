the right call for a developer-centric audience. The ActiveFilterBar chip style is contained

trip the ban (the numbers are genuine community pulse data, not vanity metrics), but it's
the first thing to reconsider if the design needs to move further from the SaaS aesthetic.

## Remaining Open Items

### [P2] No progressive filter count in the filter pane
**What**: When a user is about to check a filter option (e.g., "TypeScript"), they can't see how many results it would leave. They only know after committing the filter. The count in parentheses after each option label shows how many repos *have* that property in the full unfiltered set — not how many would remain with the current combination.
**Why it matters**: Users hesitate before applying filters they can't preview. This leads to over-filtering and then "no results" surprise.
**Fix**: Replace the static count `(72)` per option with a dynamic count showing how many results would remain if that filter were added to the current set. Requires running `filterItems()` speculatively per option — expensive for large facets (Topics has 100+ options), so apply only to the currently-visible 10 per accordion section.
**→ complex; recommended for a dedicated `/impeccable craft` pass**

### [P2] Filter facet "Focus Areas" has no descriptive context
**What**: "Focus Areas" as an accordion label is abstract. New visitors don't know if this means product area, technical domain, or something else. The facet values (e.g., "DevOps", "Governance") are meaningful but the container label isn't.
**Fix**: Add a one-line subtitle below the accordion header: "Technical domains and product areas covered by each project." — or rename the accordion to "Domains" which is clearer.
**→ `/impeccable clarify`**

### [P2] No "/" shortcut discoverability hint on the search bar
**What**: The "/" shortcut works, but users don't know it exists without trying. There's no visual hint in or around the search bar.
**Why it matters**: Keyboard shortcuts that aren't discoverable are used only by power users who already know the convention.
**Fix**: Add a small `<kbd>/</kbd>` badge inside `contentBefore` of the search Input — visible when the input is empty and not focused, disappears when the user starts typing or focuses the field.
**→ `/impeccable polish`**

### [P3] Stat card section still echoes the hero-metric template
**What**: Five stat cards with big numbers, uppercase labels, and a gradient hero above them is textbook SaaS metrics dashboard. The personas-first section order and "Ecosystem at a Glance" naming mitigate this, but the visual pattern persists.
**Why it matters**: On second-order slop test: "community hub that's not SaaS-cream → this looks like a SaaS metrics dashboard". The numbers are genuine (real repo counts, real contributors) but the presentation pattern is the tell.
**Fix options**: (a) Replace the uniform card grid with a scrolling ticker strip — numbers scroll past like a live feed rather than sitting static in cards. (b) Convert one or two stats to a text callout or prose sentence embedded in the hero subtitle. (c) Remove the stat section entirely and surface the numbers inside the gallery header ("312 repositories across 14 topics").
**→ priority low; site is functional and branded — this is an aesthetic refinement**

### [P3] Help and Documentation gap (H10 = 2)
**What**: No contextual help anywhere in the filter UI. "Good First Issue" is a GitHub label convention that first-timers may not know. "Code of Conduct" filter has no tooltip explaining why you'd filter by it.
**Fix**: Add FluentUI `Tooltip` components on the filter checkboxes for "Good First Issue", "Help Wanted", and "Code of Conduct" — short one-line explanations.
**→ `/impeccable onboard`**
