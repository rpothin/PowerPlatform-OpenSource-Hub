## Anti-Patterns Verdict

**LLM assessment**: The site largely passes the brand slop test. The committed indigo-violet gradient hero is distinctive — not the modal SaaS landing page. The persona card differentiation (variant icon tints, action links) avoids the identical-card-grid trap. The design token system is genuinely principled with zero detector violations. Two lingering concerns:

1. The "Key Insights" stats section walks up to the hero-metric template ban. Five stat cards — big number, small uppercase label, primary-colored value — is the SaaS cliché structure. 2. Stat card labels use `text-transform: uppercase; letter-spacing: 0.05em` on every card. One deliberate use is voice; five identical uppercase labels is systematic and risks reading as AI scaffolding rather than intentional design language.

## Overall Impression

The implementation is technically excellent — the token hygiene is real, the palette is committed, and the brand identity is coherent. The gap is in UX mechanics: users don't know how many results they have, they can't quickly clear search state, and there's no guidance when filters return nothing. The landing page communicates the "Community Bridge" clearly. The gallery needs feedback loops.

The single biggest opportunity: the gallery result count and empty state. These would materially improve the product experience with about 30 lines of code.

## What's Working

1. **Design token system** — Zero literal colors in any CSS file. `--ds-*` namespace is consistent, dark mode overrides are complete, and every shadow, radius, and transition pulls from the token sheet. This is the foundation that makes everything else trustworthy.

2. **Persona card differentiation** — The three personas avoid the identical-card-grid trap. The `--maintainer` (indigo tint), `--contributor` (success-green tint), and `--user` (neutral) icon wrap variants create genuine visual differentiation without violating the border-stripe ban. The action links close each card with clear intent.

3. **Gallery filter UX mechanics** — URL state persistence (filters survive browser back), mobile dialog for filters with focus return, clear-all button, and accordion sections for filter groups. This is solid information architecture for a data-dense tool.

## Priority Issues

### [P1] Stat card label contrast failure
**What**: `.statLabel` uses `var(--ds-ink-muted)` (`#7A7E9A`) at 0.72rem (11.5px) / 600 weight on a white card surface. Measured contrast: ~3.98:1.
**Why it matters**: WCAG AA requires 4.5:1 for text under 14px bold (or 18px regular). At 11.5px, even at 600 weight, this fails. DESIGN.md itself flags ink-muted: "Always checked against 4.5:1 minimum" — and this violates that promise on the most prominent surface of the landing page.
**Fix**: Darken the stat label color to `var(--ds-ink-secondary)` (`#4A4D6A`, ~8:1 contrast), or increase the font size to ≥14px where 600 weight qualifies as bold (3:1 threshold). Alternatively, increase the opacity of uppercase labels to hit the 4.5:1 floor.
**Suggested command**: `/impeccable audit`

### [P1] Gallery FluentUI dropdown clipping
**What**: `.filterPaneAndGallery` has `overflow-x: hidden`. FluentUI's Combobox components (used in the gallery for category/topic pickers) render dropdowns via `position: absolute`. Inside an `overflow: hidden` container, these dropdowns are clipped at the container boundary.
**Why it matters**: This is exactly the interaction anti-pattern flagged in SKILL.md: "Dropdowns rendered with `position: absolute` inside an `overflow: hidden` or `overflow: auto` container will be clipped." On narrower viewports, filter dropdowns that extend beyond the container height become inaccessible — users can't reach all options.
**Fix**: Remove `overflow-x: hidden` from `.filterPaneAndGallery` and solve any horizontal overflow that originally motivated it differently (e.g., `contain: layout` or fixing the child that overflows). FluentUI dropdowns will escape correctly.
**Suggested command**: `/impeccable audit`

### [P1] Gallery card `max-height: 400px` clips content
**What**: `.galleryItem` has `max-height: 400px` with no `overflow` property set. For repos with many badges/topics, card content can visually overflow the boundary with no truncation affordance.
**Why it matters**: Content overflow without explicit handling produces broken-looking cards — text or badges hanging outside the card boundary, overlapping the next card, or cut off mid-element. Riley (stress tester) will hit this immediately on any repo with >5 topics.
**Fix**: Either remove the max-height (cards are in a grid; the grid handles height distribution) or add `overflow: hidden` with a visual affordance (gradient fade, "more" indicator) to signal clipped content.
**Suggested command**: `/impeccable harden`

### [P2] Gallery page coupled to landing page module CSS
**What**: `gallery.tsx` imports `styles from './index.module.css'`. The gallery hero, layout, and filter pane classes are pulled from the landing page's CSS module.
**Why it matters**: Any edit to `index.module.css` for landing page reasons can unintentionally break the gallery — and vice versa. The coupling is invisible; there's no indication in either file that they share styles. This is a maintenance trap.
**Fix**: Create `gallery.module.css` with the gallery-specific classes extracted from `index.module.css`. Keep shared classes (like `.heroBanner` gradient) in `custom.css` or a shared module.
**Suggested command**: `/impeccable harden`

### [P2] No empty state for the gallery
**What**: When filter combinations produce zero matching repositories, the gallery renders an empty grid with no message, no illustration, and no suggested action.
**Why it matters**: Empty states are the highest-priority UX gap in discovery tools. A user who applies three filters and sees nothing has no way to know if they applied them wrong, if the data is stale, or if no repos genuinely match. They abandon — and they don't know why.
**Fix**: Add an empty state component that (1) confirms no results for the current filter set, (2) shows which filters are active, and (3) offers a "Clear all filters" CTA. Per PRODUCT.md's design principles, "Trust through clarity."
**Suggested command**: `/impeccable onboard`

## Persona Red Flags

**Alex (Power Contributor)** — arrives via the gallery link from the landing page, wants to find open issues in Power Apps repos with Go or TypeScript:
- No result count ("147 repositories" / "showing 12 of 147") — no way to know if filters are working or if the dataset is tiny.
- No keyboard shortcut to focus the search bar from anywhere on the page.
- Filter accordions are all collapsed by default; Alex must open each one to discover available options.
- No URL feedback that filters have been applied (URL updates correctly, but no visual summary of active filters in the gallery header).

**Sam (Keyboard/Screen Reader user)** — navigates the landing page with a keyboard:
- Stat card labels (0.72rem at 600 weight, `#7A7E9A`) fail 4.5:1 contrast — low-vision users relying on color contrast cannot read them.
- `focus-visible` outline is defined globally in `custom.css`, but FluentUI components override focus styles with their own Fluent ring — the indigo-violet outline may not appear consistently inside the `<FluentProvider>` boundary on the gallery page.
- The gallery `overflow-x: hidden` container may trap keyboard focus inside a dropdown that's visually clipped.

**Jordan (First-Timer, Power Platform maker new to open-source)** — lands on the home page from a Microsoft community post, wants to find a tool to extend Power Apps:
- "Key Insights" stats section appears before any explanation of what the site actually contains — Jordan sees numbers with no context.
- "Active in Last 30 Days" and "Stars+Watchers Since Last Snapshot" require knowing what these metrics mean before they're useful.
- After clicking "Browse the gallery," the gallery page hero repeats the site name and tagline (already seen) before showing any actual content. The search bar is helpful but there's no "start here" guidance — no popular categories, no featured repos.

## Minor Observations

- `heroTitle` and `heroSubtitle` use `!important` overrides to fight Docusaurus's built-in hero class styles. This is a maintenance hazard — consider replacing `className='hero hero--primary'` on the gallery page header with a custom class that doesn't apply Infima defaults.
- `sectionTitle` (max 1.5rem) renders smaller than `personasSectionTitle` (max 2rem) — the stats section looks visually less important than the personas section, which is likely unintentional.
- Dialog styles (`.dialogSectionTitle`, `.dialogDetailsLabel`) use `var(--ifm-color-secondary-darkest)` instead of `--ds-*` tokens, which means they don't benefit from dark-mode token overrides.
- `personaIconWrap--contributor` uses a literal `rgba(45, 164, 78, 0.1)` instead of a `--ds-*` token. Add `--ds-success-tint` to `custom.css`.
- The gallery header (`header` element) wraps the search bar — semantically the search bar isn't a header-level element. Consider a more neutral container.
