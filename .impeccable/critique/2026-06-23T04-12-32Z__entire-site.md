## Anti-Patterns Verdict

**LLM assessment:** The interface successfully avoids AI slop tells. No gradient text, no side-stripe borders, no cream/sand body background, no numbered eyebrows, no hero-metric template. ## Overall Impression

The site has crossed from "polished but passive" into "active decision aid." The progressive filter counts are the standout feature of this session — they turn a static checkbox list into a live negotiation where every option tells you what it costs. Combined with the active filter chips, visible keyboard shortcut, and consistent chip labels, the gallery page now rivals professional search tools in UX quality. Score 34/40 lands at the upper edge of "Good." Two more targeted improvements clear Excellent.

## What's Working

1. **Progressive filter counts** — the best design decision in this session. When a user sees "TypeScript (47)" next to "Python (12)", they're no longer guessing; they're deciding. This turns selection confidence from recall into recognition.

2. **Active filter visibility stack** — three layers working together: chip bar above results (current state), progressive counts in the pane (available transitions), and visible `/` badge in search (power shortcut). Each layer answers a different "where am I?" question.

3. **Chip label consistency** — the formatFacetLabel fix was less visible than the feature work but arguably more impactful for trust. "copilot-studio" as a chip was a credibility crack; "Copilot Studio" is seamless. Small fix, large feel.

## Priority Issues

**[P2] H10 — No contextual help for filter sections**
- **What**: "Contribution Signals" has no tooltip or subtitle explaining what it contains. "What It Covers" is clearer than "Focus Areas" was, but new users still can't predict what taxonomy a section uses until they expand it.
- **Why it matters**: The filter pane has 8 sections. Without hints, users must expand every accordion to understand the full option space — extraneous cognitive load.
- **Fix**: Add a subtitle or `aria-description` to each accordion header: e.g. "Contribution Signals — good-first-issue, help-wanted, code of conduct"; "What It Covers — data, AI, automation, etc." Alternatively, show a count of options in the collapsed header: "Topics (12)".
- **Suggested command**: `/impeccable clarify filter section headers`

**[P2] H9 — Empty state is generic; no smart recovery path**
- **What**: When filter combinations yield 0 results, the empty state says "try removing a filter" — correct but vague. Users already have progressive counts as a prevention tool, but once they've landed on 0, guidance stops.
- **Why it matters**: The progressive counts show how many results each option would give _before_ selecting. But they don't help when the zero state arrives from multiple simultaneous selections crossing each other. The empty state is the last line of recovery.
- **Fix**: Use the same `filterItems` logic to identify which single-filter removal would unlock the most results. Show: "Removing 'TypeScript' would show 14 repositories." One targeted suggestion beats "try removing a filter."
- **Suggested command**: `/impeccable harden empty state smart recovery`

**[P2] H2 — "Contribution Signals" is the last jargon label**
- **What**: The accordion is labeled "Contribution Signals" — developer-speak for boolean flags (good-first-issue, help-wanted, code of conduct). Non-developer Power Platform Makers may not parse this.
- **Why it matters**: The three audience personas include "Users" — people using Power Platform tools who want to contribute casually. For them, "Contribution Signals" is opaque.
- **Fix**: Rename to "Looking to Contribute?" or "Ways to Help" — conversational, action-oriented, matches the "What It Covers" / "Who It Helps" pattern already established.
- **Suggested command**: `/impeccable clarify contribution signals label`

**[P2] H8 — Filter pane section density approaches the edge**
- **What**: 8 accordion sections in the filter pane. Three are pre-expanded by default. On a 1080px display, users see partial accordions and must scroll within the pane to reach Licenses, Owners, Categories, etc.
- **Why it matters**: Working memory allows ≤4 items per group. 8 sections forces users to mentally track a longer structure — the pane doesn't feel scannable at a glance.
- **Fix**: Consider grouping into 2 super-sections with visual dividers: "What It Does" (Topics, What It Covers, Audiences) and "Who Made It" (Owner, License) and "Get Involved" (Contribution Signals, Languages). Visual grouping reduces perceived complexity without collapsing the UX.
- **Suggested command**: `/impeccable layout filter pane grouping`

## Persona Red Flags

**Alex (Power Developer — power user)**: Keyboard-first navigation works well — `/` shortcut is now visible, URL state is shareable. Gap: there's no keyboard shortcut to move focus to the first filter accordion (e.g., `F` to open filter pane). Tab + Enter gets there but requires multiple key presses. Low risk for a dedicated power user; worth noting.

**Jordan (First-Time Contributor)**: Lands on the gallery, sees 8 accordion sections. Three are open by default (good). But Licenses, Owners, Categories, Focus Areas, and Audiences are collapsed with no hint of what they contain. Jordan may not know "Contribution Signals" means "filters for repos asking for help." The "Good First Issue" label only appears after expanding the accordion — not surfaced at the category level. High abandonment risk for the contributing journey.

**Sam (Power Platform Maker — non-developer)**: The landing page persona card for "Users" links to `/gallery` with clear intent. Sam arrives at the gallery and sees "Contribution Signals" — jargon that reads as developer-internal. If Sam just wants tools for Power Automate, they must know to look in "What It Covers" → Power Automate. The section label change from "Focus Areas" to "What It Covers" helps, but the accordion still requires expansion to reveal the options.

## Minor Observations

- The stat row on the landing page shows up to 5 cards when momentum data is present. Five side-by-side metrics pushes the working memory limit (Miller: ≤4). The momentum card could be deprioritized or shown as a footnote.
- Gallery hero heading uses `siteConfig.title` = "Power Platform Open-Source Hub" — redundant since the user is already on the gallery page. A focused heading like "Browse Community Repositories" would reduce repetition and orient the user to the task.
- `ActiveFilterBar` "Clear all" is conditionally rendered for `chips.length > 1`. With exactly 1 active filter, no "Clear all" appears. This is technically correct UX (the chip itself has a dismiss) but could briefly confuse users expecting a consistent control.
- No `lang` attribute variation or RTL consideration — acceptable for this audience but worth noting for future internationalisation.

## Questions to Consider

- "What It Covers" and "Categories" both sound like taxonomy labels — is the distinction between them clear enough from the filter pane alone, or does the overlap create confusion when all 8 accordions are visible at once?
- If the filter pane had a "Quick filters" row above the accordions — 3-4 the most-used filter buttons — would power users reach their goal faster without opening any accordion?
- Does the landing page stat row communicate _value_ or just _scale_? A user seeing "434 Tracked Repositories" doesn't know if that's impressive or expected. What would make those numbers feel meaningful?
